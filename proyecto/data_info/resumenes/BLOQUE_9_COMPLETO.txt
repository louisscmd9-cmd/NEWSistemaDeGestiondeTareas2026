📊 CAMBIOS EN BASE DE DATOS
═════════════════════════════════════════════════════════════════════════════════

Tabla: tasks

ANTES:
  CREATE TABLE tasks (
    id INTEGER PRIMARY KEY,
    titulo TEXT,
    descripcion TEXT,
    estado TEXT,
    fecha TEXT,
    id_empleado INTEGER,
    explicacion TEXT
  );

DESPUÉS:
  CREATE TABLE tasks (
    id INTEGER PRIMARY KEY,
    titulo TEXT,
    descripcion TEXT,
    estado TEXT,
    fecha TEXT,
    id_empleado INTEGER,
    explicacion TEXT,
    id_jefe INTEGER NOT NULL,           ← NUEVO
    recurrente INTEGER DEFAULT 0,       ← NUEVO
    fecha_creacion DATETIME DEFAULT CURRENT_TIMESTAMP,  ← NUEVO
    FOREIGN KEY (id_empleado) REFERENCES users(id),
    FOREIGN KEY (id_jefe) REFERENCES users(id)         ← NUEVO
  );

  INDEX idx_tasks_id_jefe ON tasks(id_jefe);        ← NUEVO
  INDEX idx_tasks_recurrente ON tasks(recurrente);  ← NUEVO

🎓 EJEMPLOS DE USO

Caso 1: Jefe crea reunión diaria
  Lunes:
    Jefe crea: "Reunión 9 AM"
    Marca: "Tarea Recurrente"
  
  Martes (automático):
    Script crea copia para hoy
    Misma descripción, empleado
    Estado: pendiente
  
  Miércoles (automático):
    Otro día, otra copia
  
  Si jefe necesita cambiar hora:
    Click "Editar"
    Modal abre
    Cambia a "9:30 AM"
    Guardar
    ✓ Se actualiza solo para ese día

Caso 2: Jefe necesita cancelar
  Vio que reunión no va más:
    Click "Eliminar"
    Confirma
    ✓ Se elimina solo esa instancia
    ✓ Mañana se creará otra (recurrencia sigue)
  
  Vio que nunca más hay reunión:
    Click "Editar"
    En modal, desmarcar "Tarea Recurrente"
    Guardar
    ✓ Se deja de duplicar
    ✓ Pero hoy sigue activa

Caso 3: Asunto crítico surge
  Jefe modificó prioridad de empleado:
    Click "Editar"
    Cambiar "Empleado" de "Carlos" a "Ana"
    Guardar
    ✓ Se asigna ahora a Ana
    ✓ Mañana se copia también para Ana
